# **ROS 通用核心概念（ROS1 & ROS2）**

  

> 只整理 **ROS1 和 ROS2 都存在/通用的概念**，每个小节最后用一行简单对比实现差异，方便你以后单独做“差异笔记”。

---

## **1. 节点（Node）**

- **定义**
    
    - 一个可执行程序，就是一个节点。
        
    - 多个节点组成一个分布式系统，共同完成机器人功能。
        
    
- **特点**
    
    - 节点之间不直接调用函数，而是通过通信机制（话题、服务、动作等）交互。
        
    - 节点名在整套系统中应唯一。
        
    - 节点可以位于不同机器上（同一网络）。
        
    
- **常见实践**
    
    - 将系统拆成多个小节点，而不是一个超级大节点。
        
    - 每个节点关注一件事：感知、控制、规划、界面等。
        
    
- **ROS1 vs ROS2**
    
    - 通用思想完全一样；差异主要在底层中间件和启动方式。
        
    

---

## **2. 通信模型总览**
### **2.1 发布者 / 订阅者（Pub / Sub）**

- **定义**
    
    - 发布者（Publisher）：向某个话题发送消息。
        
    - 订阅者（Subscriber）：订阅某个话题，接收消息。
        
    
- **特点**
    
    - **多对多、弱耦合**：发布者并不知道谁在订阅，反之亦然。
        
    - 常用于 **连续数据流**：传感器数据、控制指令等。
        
    

  

### **2.2 客户端 / 服务器（Client / Server）**

- **定义**
    
    - 服务服务器：提供某个请求–响应接口。
        
    - 服务客户端：发起请求，并等待响应。
        
    
- **特点**
    
    - **同步调用**：类似函数调用，请求一次，返回一次。
        
    - 适合 **一次性任务**：获取当前状态、触发某个动作等。
        
    
- **ROS1 vs ROS2**
    
    - Pub/Sub、Client/Server 模型概念完全一致，只是具体 API 和底层实现不同。
        
    

---

## **3. 话题（Topic）与消息（Message）**

  

### **3.1 话题（Topic）**

- **定义**
    
    - 一个带名字的“数据通道”，通过它传输某种类型的消息。
        
    
- **特点**
    
    - 每个话题绑定一种消息类型（Msg）。
        
    - 典型命名：/cmd_vel、/scan、/odom 等。
        
    

  

### **3.2 消息（Msg）**

- **定义**
    
    - 话题中传输的数据结构；
        
    - 由一组字段组成（标量 + 数组 + 嵌套类型）。
        
    
- **定义方式（通用）**
    
    - 使用 .msg 文件声明：
```
float32 x
float32 y
float32 z
```
- -   
        
    - 编译时自动生成相应语言的类型（C++、Python 等）。
        
    
- **ROS1 vs ROS2**
    
    - Topic & Msg 概念完全通用；.msg 文件语法基本相同。
        
    

---

## **4. 服务（Service）与服务消息（Srv）**

- **Service 概念**
    
    - 一次请求，一次响应。
        
    - 适合：
        
        - 获取当前配置/状态；
            
        - 发出一次性指令（如重置里程计）。
            
        
    
- **Srv 文件结构**
    
    - .srv 文件由两部分组成，中间用 --- 分隔：
```
int32 a
int32 b
---
int32 sum
```
- **使用场景**
    
    - “要立即知道结果”的操作；
        
    - 不适合长时间执行任务（这类任务更适合 Action）。
        
    
- **ROS1 vs ROS2**
    
    - Service 模型、.srv 文件结构均通用。
        
    

---

## **5. 动作（Action）**

- **解决的问题**
    
    - 服务是同步等待，遇到“执行时间较长”的任务就不合适：
        
        - 例如：走到一个目标点、执行复杂动作等。
            
        
    
- **Action 特点**
    
    - 支持：
        
        - 发送目标（Goal）
            
        - 反馈（Feedback）
            
        - 结果（Result）
            
        - 取消（Cancel）
            
        
    - Action Server / Action Client 通信。
        
    
- **Action 定义文件**
    
    - .action 文件同样用 --- 分三部分：
```
# Goal
geometry_msgs/Pose target_pose
---
# Result
bool success
---
# Feedback
float32 progress
```
-   
    
- **ROS1 vs ROS2**
    
    - 概念完全一致，只是库和接口略有差异。
        
    

---

## **6. 参数（Parameter）**

- **定义**
    
    - 一些配置数据，通过 **参数服务器 / 参数系统** 存储。
        
    
- **特点**
    
    - 典型用于：PID 参数、话题名、帧名、阈值等配置项。
        
    - 一般有命名空间结构：/controller/kp、/robot/description 等。
        
    - 节点可在启动时读取，或在运行时动态改变。
        
    
- **好处**
    
    - 代码与配置解耦；
        
    - 不需要重新编译即可调整参数。
        
    
- **ROS1 vs ROS2**
    
    - “有参数服务器 / 参数系统”这一概念通用；
        
    - 访问方式与作用域管理方式有所不同。
        
    

---

## **7. 包（Package）**

- **定义**
    
    - ROS 软件组织的最小单元：包含节点源码、消息定义、配置、launch 文件等。
        
    
- **结构（通用概念）**
    
    - src/ 源码
        
    - msg/, srv/, action/ 类型定义
        
    - launch/ 启动文件
        
    - CMakeLists.txt, package.xml 构建与依赖描述
        
    
- **包的职责**
    
    - 描述自己依赖哪些其他包；
        
    - 提供可执行文件和接口。
        
    
- **ROS1 vs ROS2**
    
    - Package、package.xml 等概念通用；
        
    - 构建工具不同（Catkin vs ament/colcon），但“包”的思想相同。
        
    

---

## **8. 工作空间（Workspace）**

- **定义**
    
    - 一块目录区域，用来放多个 ROS 包并统一构建与管理。
        
    
- **典型结构（概念）**
    
    - src/：存放所有包的源码。
        
    - build/：编译产生的中间文件。
        
    - install/ 或 devel/：最终可运行文件、脚本、环境设置。
        
    
- **使用方式**
    
    - 在 workspace 根目录执行构建命令；
        
    - 用 setup.bash（或类似脚本）加载环境，使系统识别这些包。
        
    
- **ROS1 vs ROS2**
    
    - “有工作空间 + 统一构建 + source 环境”的概念通用。
        
    - 区别主要是目录名和构建命令。
        
    

---

## **9. 构建系统 & 依赖管理**

- **通用思路**
    
    - 用 CMake + 元工具（catkin/ament）来：
        
        - 编译 C++ 节点；
            
        - 生成消息/服务/动作代码；
            
        - 处理依赖关系（其他包、系统库等）。
            
        
    
- **依赖声明**
    
    - 通过 package.xml 声明依赖包。
        
    - 构建系统根据这些依赖自动准备 include 路径、链接库等。
        
    
- **ROS1 vs ROS2**
    
    - “CMake 驱动 + package.xml 声明依赖”的概念通用；
        
    - 具体工具名不同（catkin vs ament）。
        
    

---

## **10. 启动系统（Launch）**

- **目标**
    
    - 用一个配置文件一次性启动多节点，并设置参数、重映射话题等。
        
    
- **典型功能**
    
    - 同时启动多个节点；
        
    - 设定命名空间；
        
    - 设置参数；
        
    - 话题重映射（rename topic）；
        
    - 包含其他 launch 文件（复用配置）。
        
    
- **常见使用场景**
    
    - 机器人整机启动；
        
    - 仿真环境启动（Gazebo / RViz + 一系列节点）。
        
    
- **ROS1 vs ROS2**
    
    - “写一个 Launch 文件来管理整个系统”的思想完全一致；
        
    - 只是语法和格式（XML / Python 等）不同。
        
    

---

## **11. 坐标变换（TF 框架）**

- **核心问题**
    
    - 机器人各个部件（底盘、激光雷达、相机、机械臂关节等）都有坐标系；
        
    - 需要在这些坐标系之间进行转换（位置、姿态）。
        
    
- **TF 提供的能力**
    
    - 发布坐标变换（frame A → frame B）；
        
    - 查询任意两坐标系之间的变换；
        
    - 随时间变化的变换（时间戳）。
        
    
- **常见坐标系**
    
    - map、odom、base_link、laser、camera_link 等。
        
    
- **ROS1 vs ROS2**
    
    - TF / TF2 模型概念通用（坐标树、变换缓冲区等）。
        
    

---

## **12. 数据录制与回放（Bag）**

- **作用**
    
    - 把话题数据录下来，之后可以离线回放进行调试/仿真。
        
    
- **常见用途**
    
    - 现场采集真实传感器数据，在实验室重现问题；
        
    - 无需真实机器人即可反复测试算法。
        
    
- **基本概念**
    
    - Bag 文件：一个包含多话题、多时间戳消息的日志文件。
        
    - 录制：指定要记录的话题。
        
    - 回放：按时间顺序重新发布这些话题消息。
        
    
- **ROS1 vs ROS2**
    
    - 有 rosbag / ros2 bag，但“录制话题数据再回放”的概念完全相同。

---

## **13. 消息接口文件：.msg/.srv/.action**


- **通用设计思想**
    
    - 通信接口（数据格式）用简单文本文件定义；
        
    - 构建时自动生成各语言的代码；
        
    - 实现与接口分离，方便复用与跨语言。
    
- **三种文件**
    
    - .msg：话题消息格式。
        
    - .srv：服务请求/响应格式。
        
    - .action：动作接口（Goal / Result / Feedback）。
        
    
- **好处**
    
    - 清晰描述节点之间的“契约”；
        
    - 当接口稳定后，可以方便更换底层实现。
        
    
- **ROS1 vs ROS2**
    
    - 三类接口文件的 **存在形式和用途** 均通用。
        
    

---

## **14. 分布式系统与中间件（高层概念）**

- **分布式**
    
    - 节点可以在多台机器上运行；
        
    - 使用通信中间件来发现彼此、传输消息。
        
    
- **中间件职责（抽象层面）**
    
    - 节点发现；
        
    - 话题匹配；
        
    - 数据序列化/反序列化；
        
    - 传输可靠性/实时性管理。
        
    
- **对开发者的影响**
    
    - 在应用层，依然是 Pub/Sub、Service、Action 这些熟悉的概念；
        
    - 底层中间件不同，但高层抽象保持一致。
        
    
- **ROS1 vs ROS2**
    
    - 都是“基于中间件的分布式系统”；
        
    - 区别是使用的中间件种类不同
